🔒 Singleton

O Problema:

Em um sistema, às vezes precisamos de APENAS UMA instância de uma classe.

  • 📁 Conexão com banco de dados - abrir 100 conexões trava o servidor
  • 📝 Sistema de Logs - todos os logs devem ir para o mesmo arquivo
  • ⚙️ Configurações globais - todo mundo usa as mesmas configurações
  • 🖨️ Fila de impressão - só uma fila controla todos os documentos

⚠️ Sem Singleton: Cada parte do programa cria sua própria conexão → Bagunça e desperdício!

Como o Singleton resolve?

❌
Conexão 1
Conexão 2
Conexão 3

Sem Singleton: Várias instâncias

→
✅
ÚNICA Conexão

Com Singleton: Uma instância compartilhada

Regra de ouro: O construtor é privado e um método estático retorna sempre a MESMA instância.

🏭 Factory Method

O Problema:

Seu código precisa criar muitos tipos diferentes de objetos (ex: EmailService, SMSService, PushService).

// SEM Factory - código bagunçado e repetido
if (tipo == "email") service = new EmailService();
if (tipo == "sms") service = new SMSService();
if (tipo == "push") service = new PushService();
// Imagine isso em 10 lugares diferentes...

Problema: O código cliente fica acoplado às classes concretas. Se um novo tipo surgir, você precisa mudar em vários lugares!

Como o Factory resolve?

✅ Centralizamos a criação em uma FÁBRICA!

// COM Factory - código limpo e organizado
class NotificationFactory:
    def create(type):
        if type == "email": return EmailService()
        if type == "sms": return SMSService()

// No código cliente:
service = NotificationFactory.create("email")

Vantagem: Se surgir um novo tipo (ex: WhatsAppService), só a Factory muda. O resto do código continua igual!

💻 Código: Singleton em Python

# Exemplo: Conexão com Banco de Dados (única instância)
class DatabaseConnection:
    _instance = None   # Guarda a única instância

    def __new__(cls):
        if cls._instance is None:
            print("🔌 Criando conexão com o banco...")
            cls._instance = super().__new__(cls)
        return cls._instance

    def query(self, sql):
        print(f"📀 Executando: {sql}")

# Testando
db1 = DatabaseConnection()
db2 = DatabaseConnection()
print(f"São iguais? {db1 is db2}")  # True!

🔑 Mágica: O método __new__ controla a criação. Se já existe, reutiliza!

💻 Código: Factory em Python

# Produtos (diferentes tipos de serviço)
class EmailService:
    def send(self, msg): return f"📧 Email: {msg}"

class SMSService:
    def send(self, msg): return f"📱 SMS: {msg}"

class PushService:
    def send(self, msg): return f"🔔 Push: {msg}"

# A FÁBRICA
class NotificationFactory:
    def create(self, tipo):
        if tipo == "email": return EmailService()
        if tipo == "sms": return SMSService()
        if tipo == "push": return PushService()

# Usando a fábrica
fabrica = NotificationFactory()
servico = fabrica.create("email")
print(servico.send("Bem-vindo!"))

🎯 Juntando tudo!

# 1. SINGLETON - Logger global
class Logger:
    _instance = None
    
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.logs = []
            print("📝 Logger criado!")
        return cls._instance
    
    def log(self, msg):
        self.logs.append(msg)
        print(f"[LOG] {msg}")

# 2. PRODUTOS (Serviços de notificação)
class EmailService:
    def send(self, msg):
        Logger().log(f"Email enviado: {msg}")
        return f"📧 {msg}"

class SMSService:
    def send(self, msg):
        Logger().log(f"SMS enviado: {msg}")
        return f"📱 {msg}"

# 3. FACTORY
class NotificationFactory:
    def create(self, tipo):
        if tipo == "email": return EmailService()
        if tipo == "sms": return SMSService()

# 4. EXECUTANDO
fabrica = NotificationFactory()
servico1 = fabrica.create("email")
servico2 = fabrica.create("sms")

print(servico1.send("Olá mundo!"))
print(servico2.send("Mensagem urgente"))

# O mesmo logger foi usado nas duas chamadas!

❓ Perguntas da aula

1. PARA QUE SERVE O SINGLETON EM UM SERVIÇO?

Garante que um recurso compartilhado (conexão com banco, logger, cache) tenha UMA ÚNICA INSTÂNCIA em todo o servidor. Evita desperdício de memória, conflitos e inconsistências.

2. COMO O FACTORY AJUDA COM MUITOS TIPOS DE SERVIÇOS?

Centraliza a lógica de criação em um único lugar. Se um novo tipo surgir, você só muda a Factory. O código cliente fica desacoplado e mais fácil de dar manutenção.

3. POR QUE CONTROLAR A CRIAÇÃO DE OBJETOS NO SERVIDOR?

Para garantir que objetos sejam criados corretamente (com conexões abertas, configurados), evitar vazamento de memória, aplicar políticas (ex: limite de instâncias) e facilitar testes.
